## Janus86をコアにした可逆エフェクト実行環境

## 1. 概要

**UNCALL**は、可逆プログラミング言語Janus86をコアに、プログラム内の計算だけでなく、リソース作成、クリーンアップ、ロールバック、量子ゲートの逆演算といった操作を、`call`と`uncall`の対として記述・実行するためのブラウザ実行環境である。

通常のプログラムでは、処理を取り消すためのcleanupコードやrollbackコードを、順方向の処理とは別に人間が管理する必要がある。UNCALLでは、各プリミティブに順方向と逆方向の動作を定義し、プログラムの構造から逆実行順序を自動的に導出する。

```janus
procedure deploy()
    call create_network(network)
    call create_database(database)
    call deploy_application(application)
```

このprocedureを逆方向に実行すると、同じ定義から次の処理が得られる。

```text
undeploy_application
delete_database
delete_network
```

UNCALLは「実行履歴を巻き戻すデバッガ」ではない。

**順方向の処理と、それに対応する逆方向の処理を、最初から同じプログラムとして設計するための実験的ランタイム**である。

---

## 2. 一言で表すと

> Janus86の`call`／`uncall`を、実用プログラムのcleanup、rollback、uncomputationへ拡張する。

---

## 3. 背景

プログラムには、順方向の処理と対になる後処理が数多く存在する。

* リソースを作成する／削除する
* ファイルを開く／閉じる
* メモリを確保する／解放する
* トランザクションを開始する／取り消す
* 一時環境を構築する／破棄する
* 暗号化する／復号する
* 量子ゲートを適用する／随伴ゲートを適用する
* 中間値を計算する／uncomputeする

しかし、現在の一般的なプログラミングでは、これらは別々のコードとして記述される。

```ts
await createNetwork();
await createDatabase();
await deployApplication();

// 別の場所にcleanupを書く
await undeployApplication();
await deleteDatabase();
await deleteNetwork();
```

その結果、次の問題が発生する。

* cleanupの実行順序を間違える
* 新しい処理を追加してもcleanupを更新し忘れる
* 一部成功後の失敗状態を正しく処理できない
* rollbackコードが通常経路でテストされない
* 作成処理と削除処理の対応関係がコードから見えない
* 厳密な逆操作と、単なる補償処理が混同される

可逆プログラミングでは、順方向のプログラムを反転すると、文の順序が逆になり、各操作が対応する逆操作へ置き換わる。

UNCALLはこの構造を、Janus86の小さな言語コアとTypeScriptホストAPIによって実用プログラムへ接続する。

---

## 4. 解決方法

UNCALLは二層で構成する。

### Janus86コア

Janus側では、処理の順序、条件分岐、可逆更新、procedure、`call`／`uncall`を記述する。

```janus
procedure provision()
    call create_network(network, network_receipt)
    call create_database(database, database_receipt)
    call deploy_application(application, application_receipt)
```

### TypeScriptホスト

外部システムへ影響を与えるプリミティブはTypeScript側で登録する。

```ts
runtime.registerPrimitive("create_network", {
  reversibility: "checked",

  async forward({ config }) {
    const resource = await cloud.createNetwork(config);

    return {
      receipt: {
        resourceId: resource.id,
        generation: resource.generation,
      },
    };
  },

  async backward({ receipt }) {
    await cloud.deleteNetwork(receipt.resourceId);
  },
});
```

Janusは逆実行の構造を管理し、TypeScriptは実際のAPI、認証、retry、timeout、永続化を担当する。

## 5. 重要な設計思想

### 5.1 何でも可逆だとは扱わない

順方向と逆方向の関数を書くだけでは、すべての操作が数学的に可逆になるわけではない。

UNCALLでは、プリミティブを次の4種類に分類する。

| 種類             | 意味                 | 例                          |
| -------------- | ------------------ | -------------------------- |
| `exact`        | 数学的に厳密な逆操作         | XOR、swap、量子unitary gate    |
| `checked`      | 事前・事後条件が成立する場合に逆操作 | 空ディレクトリ作成／削除、未変更リソースの作成／破棄 |
| `compensating` | 元の状態を完全には復元しない補償操作 | 課金／返金、予約／キャンセル             |
| `irreversible` | 逆方向へ戻れないcommit境界   | メール送信、measurement、外部通知     |

これにより、クラウドリソースの削除と量子ゲートの随伴演算を、同じ「逆操作」という言葉で曖昧に扱わない。

### 5.2 外部操作にはreceiptを残す

外部APIでは、作成されたリソースIDやversionなど、逆操作に必要な情報が実行時に初めて決まる。

そのためforward操作は、逆実行に必要な最小情報をreceiptとして返す。

```text
Empty receipt
    ── forward ──▶
Created receipt(resource ID, generation)
    ── backward ──▶
Empty receipt
```

receiptはコピー不能な線形リソースとして扱うことを将来目標とする。

これにより、次の誤りを検出できる。

* 同じリソースを二度削除する
* 一つのreceiptを複数のcleanupで使用する
* cleanupせずにscopeを終了する
* 別の実行で生成したreceiptを誤用する

### 5.3 逆実行をsnapshot復元にしない

UNCALLのbackward実行は、過去の変数状態を保存して戻す方式ではない。

```text
現在の状態
  ＋
現在の制御位置
  ＋
各プリミティブのbackward定義
```

から実際に逆方向の処理を実行する。

UI表示用のイベントログは保持してもよいが、実行の正しさをsnapshotに依存させない。

UNCALLは、独立したJanus言語として動作すると同時に、TypeScriptアプリケーションへ埋め込める。

### 独立したJanusプログラム

```janus

```

### TypeScriptからの利用

```ts
const module = compileJanus(source);

module.registerPrimitive("create_network", createNetworkEffect);
module.registerPrimitive("create_database", createDatabaseEffect);

await module.call("deploy", initialState);
await module.uncall("deploy", finalState);
```

### Tagged templateによる埋め込み

```ts

procedure deploy()
    call create_network(network)
    call create_database(database)
`;
```

任意のTypeScriptコードを自動的に可逆化することは目標としない。

Janusの構文と静的制約を維持しながら、ホストアプリケーションから呼び出せることを目指す。

## 7. MVPの目的

MVPでは、次の問いに動くソフトウェアで答える。

> 順方向と逆方向を持つプリミティブをJanus86に登録すれば、同じプログラム定義から実行とcleanupを生成できるか。

さらに、外部操作の種類によって、厳密な逆、条件付き逆、補償、不可能な逆を区別できることを示す。

## 9. メインデモ

### Reversible Infrastructure Sandbox

ブラウザ内に仮想的なインフラ環境を用意する。

```text
Network
├── Database
└── Application
```

Janusコードで構築手順を記述する。

```janus
procedure deploy()
    call create_network(network, network_receipt)
    call create_database(database, database_receipt)
    call deploy_application(application, application_receipt)
```

### 正常実行

```text
create_network       success
create_database      success
deploy_application   success
```

リソースグラフが順番に構築される。

その後`uncall deploy()`を実行すると、

```text
undeploy_application
delete_database
delete_network
```

の順番で削除される。

### 途中失敗

`deploy_application`を意図的に失敗させる。

```text
create_network       success
create_database      success
deploy_application   failed
```

現在のcontrol stackと生成済みreceiptに基づき、成功済みの操作だけを逆順でcleanupする。

```text
delete_database      success
delete_network       success
```

このデモにより、次を一度に示せる。

* 文列の構造的な反転
* `call`と`uncall`
* 逆順cleanup
* receipt
* 部分成功からの復旧
* 外部effectとJanus内部状態の分離
* checked reversible operation

## 10. セカンドデモ

### Quantum Gate Adjoint

同じJanusの反転構造が、量子ゲートでは数学的な随伴演算になることを示す。

```janus
procedure bell_pair()
    call H(q0)
    call CNOT(q0, q1)
```

`uncall bell_pair()`は、次の逆gate列になる。

```text
CNOT†
H†
```

HとCNOTは自己随伴なので、実際には同じgateが逆順に並ぶ。

```text
Forward  : H → CNOT
Backward : CNOT → H
```

MVPでは実機や本格的な量子シミュレータは不要とする。

* gate列
* inverse gate列
* 簡単なstate vector
* forward後にbackwardして初期状態へ戻ること

までを表示する。

このデモは、インフラcleanupと量子uncomputationが同一ではないことも明示する。

| インフラ操作         | 量子ゲート            |
| -------------- | ---------------- |
| 外部APIのrollback | unitaryのadjoint  |
| 失敗する可能性がある     | 数学的に逆が定義される      |
| receiptが必要     | gate定義から逆を生成可能   |
| 条件付きまたは補償的     | exact reversible |

## 12. 技術選択

### TypeScript

MVPでは、処理系とUIの両方をTypeScriptで実装する。

理由:

* ブラウザで直接実行できる
* TypeScriptホストAPIを同じコードベースで提供できる
* ASTや実行イベントをUIへ渡しやすい
* Web Workerへ分離できる
* mock effectの実装が容易
* npmパッケージ化できる

Rust＋WebAssemblyは、回路生成や大規模実行が必要になった段階で検討する。

### Parser

* 手書きtokenizer
* recursive descent parser
* 式はPratt parser
* 全ASTノードにsource spanとnode IDを付与

### 逆実行

ASTは不変とし、現在の実行位置をcontrol frameとして保持する。

```ts
interface MachineState {
  store: Store;
  frames: ControlFrame[];
  direction: "forward" | "backward";
  receipts: ReceiptStore;
}
```

過去のstore snapshotを逆実行の根拠にはしない。

## 17. 新規性

既存のrollbackシステムやInfrastructure as Codeは、一般に操作履歴、差分、依存グラフ、個別に実装された削除処理によって復旧を行う。

UNCALLは、Janus86のプログラム反転を中心に置き、

* procedureを反転する
* 文の順序を逆にする
* `call`を`uncall`へ変換する
* 外部primitiveにbackward interfaceを与える
* 厳密な逆と補償を型として区別する

という、プログラミング言語側からのアプローチを取る。

単なるJanus Playgroundでも、一般的なworkflow engineでもない。

> **可逆プログラムを、外部effectのlifecycleを記述する小さなcore languageとして利用する。**

これがUNCALLの中心的な提案である。

## 18. 審査員に伝えるストーリー

通常のプログラムでは、何かを作るコードと、それを片付けるコードを別々に書く。

そのため、実装が進むほどcleanupは古くなり、失敗経路で初めて壊れていることが分かる。

Janusには、procedureを逆方向に呼び出す`uncall`がある。

UNCALLは、この30年以上前の言語機能を、現在の外部effectへ接続する。

```text
call   = construct
uncall = destruct
```

ただし、現実の世界ではすべてが完全に元へ戻るわけではない。

そのためUNCALLは、

* exact inverse
* checked rollback
* compensation
* irreversible boundary

を区別する。

同じ反転構造は、インフラではcleanupとなり、量子回路ではadjointとなる。

> **何かを実行するコードを書いたとき、その逆方向の責任も同じ場所に記述する。**

それがUNCALLの提案するプログラミングモデルである。

## 20. 最終メッセージ

UNCALLは、Janus86を懐かしい言語として復刻するプロジェクトではない。

Janus86の`call`／`uncall`という単純な構造を使い、現代のプログラムにおけるcleanup、rollback、resource lifecycle、uncomputationを、一つのプログラミングモデルとして捉え直す。

> **Forwardを書くなら、Backwardも同じプログラムに存在すべきだ。**
