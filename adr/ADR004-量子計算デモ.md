# ADR004: QFTとToffoli量子加算器を別ページの2タブで提供する

## Status

Accepted — Implemented

## Current state

量子計算デモは未実装である。既存のメインページにはPure Janusの可逆計算デモが3つあり、さらに量子デモを同じ切替UIへ追加すると説明量が過大になる。

JanusからTypeScriptのhost primitiveを呼ぶ経路、primitiveのforward / backward handler、receipt journal、実行event、同一procedureからforward / backwardの順序を導出する仕組みは実装済みである。量子デモではこのうちhost primitiveの逐次実行を使い、各handlerがブラウザ側へ量子ゲートをemitする。

## Date

2026-07-18

## 概要

UNCALLに、QFTとToffoliベース量子加算器を可視化する技術デモを追加する。メインページの4つ目、5つ目のデモにはせず、`/quantum`の別ページに二つのtabとして提供する。

`QFT` tabでは、3-qubitの計算基底をQFTすると振幅の大きさが均等になり、入力値が相対位相へ現れることを、回路図とphasorで示す。backward実行では逆QFTをemitし、元の計算基底へ戻す。

`Adder` tabでは、二つの計算基底registerを入力に取り、Janus procedureをforward実行するとToffoliベースの加算回路がTypeScriptへ順番にemitされ、backward実行すると同じprocedureから減算回路が順番にemitされる様子を見せる。

二つの中心となる変換は次とする。

```text
call qft
|x⟩  →  1 / √(2^n) Σ_y exp(2πixy / 2^n)|y⟩

uncall qft
QFT|x⟩  →  |x⟩

call add
|a⟩|b⟩|0...0⟩  →  |a⟩|a + b mod 2^n⟩|0...0⟩

uncall add
|a⟩|a + b⟩|0...0⟩  →  |a⟩|b⟩|0...0⟩
```

加算器の出典として、Craig Gidneyの[Halving the cost of quantum addition](https://doi.org/10.22331/q-2018-06-18-74)（[arXiv:1709.06648](https://arxiv.org/abs/1709.06648)）をページ上に明記する。ただし、最初のデモは論理Toffoliをprimitiveとして扱い、論文のClifford+T分解、temporary logical-AND、測定ベースuncomputation、T-count最適化までは再現しない。

したがってUIでは、出典のある**可逆な量子加算器**として紹介する。`4n + O(1)`など、デモ自身が検証していないresource削減を成果として表示しない。

## ADR000〜ADR003との関係

ADR000の初期企画には量子回路の可逆実行が将来候補として含まれていた。本ADRはその候補を独立した技術デモとして具体化する。

ADR001で実装したcalls-only kernelとhost primitive接続を再利用する。量子回路専用の別言語や、表示だけの固定ゲート配列から始めない。

ADR002のPure Janusとhost effectの分離は維持する。Pure Janus evaluatorへcallbackや量子ゲートemitを追加せず、量子ゲートをhost primitiveとして扱う。procedure引数も本デモのためには追加しない。

ADR003ではquantum uncomputationを一般ユーザー向けheadlineとしては保留した。本ADRでもメインページのheadlineを量子デモへ置き換えない。`/quantum`は、QFT、可逆加算、逆回路とJanusの対応を見たい人向けの独立した技術デモとする。

## 背景

### メインページへ追加しない理由

現在のメインページは、ソート、文字列codec、木とpathのcodecを切り替えてPure Janusの可逆計算を説明している。量子回路にはqubit、ancilla、control、target、計算基底、重ね合わせ、位相、ゲート図という固有の説明が必要になる。

同じページへ追加すると、既存デモの説明と量子計算の前提説明が競合する。メインページには`Quantum demo`への短いリンクだけを置き、量子ページ内で完結させる。

### QFTと加算器を別tabにする理由

QFTは、数値が計算基底から相対位相へ移り、逆QFTで元へ戻る過程そのものが見せ場になる。意味のある可視化には複素振幅と位相表示が必要である。

Toffoliベースの加算器は、計算基底上の`X`、`CNOT`、`Toffoli`によるbitの変化が見せ場になる。両者をDraper型QFT加算器として一つにすると、位相による加算まで一度に説明する必要があり、各デモの焦点が弱くなる。

このためQFTと加算器は独立したtabにする。QFT tabは位相と逆QFT、Adder tabは論理gateの逐次emitと加減算へ集中する。二つのtabは、同じJanus host runtimeがforward / backwardのgate streamを作る点だけを共有する。

### Gidney加算器と本デモの抽象度

参照論文は、temporary logical-ANDを4個のT gateで計算し、後で追加のT gateなしに消去することで、n-bit加算器のT-countを`8n + O(1)`から`4n + O(1)`へ減らす。

この非対称性は測定を利用するためである。temporary logical-ANDを厳密に展開する場合、forwardの計算回路とbackwardの消去回路は単なる逆順の同一物理ゲート列にはならない。消去側にはX基底測定、古典測定結果に依存する補正、ancillaの再利用が現れる。

本デモの目的はfault-tolerant synthesisのresource比較ではなく、Janus procedureのdirection反転から量子加算回路と減算回路が得られることを示すことである。このため`X`、`CNOT`、`Toffoli`を分解しない論理primitiveとする。

| 項目 | 本デモで扱う | 本デモで扱わない |
| --- | --- | --- |
| 加算器の論理構造 | 扱う | — |
| 計算基底上の加算と減算 | 扱う | — |
| `X` / `CNOT` / `Toffoli`の回路図 | 扱う | — |
| ancillaの初期化とzero復帰 | 扱う | — |
| ToffoliのClifford+T分解 | — | 扱わない |
| temporary logical-ANDの測定ベース消去 | — | 扱わない |
| T-count / T-depthの再現と比較 | — | 扱わない |
| 実量子hardwareでの実行 | — | 扱わない |

## 決定

## 1. デモは`/quantum`に置き、二つのtabに分ける

Cloudflare Workerへ次のrouteを追加する。

```text
GET /quantum          QFTと量子加算器デモのHTML
GET /quantum/app.js   量子デモ専用browser bundle
```

既存の`/`と`/app.js`の責務は変えない。デザインtokenとbrand表現は共有してよいが、量子回路固有のCSSとbrowser制御は別moduleに分ける。

メインページには別ページへのリンクだけを追加する。既存のdemo tab、source editor、状態管理へ量子デモを混在させない。

量子ページ内には次の二つのtabを置く。

| Tab | 主題 | 入力 | Forward | Backward |
| --- | --- | --- | --- | --- |
| QFT | 計算基底の値が相対位相へ移る | 3-bitの計算基底`|x⟩` | QFT | 逆QFTで`|x⟩`へ戻す |
| Adder | Toffoliベースの可逆加算 | 4-bitの`|a⟩|b⟩` | `b += a mod 16` | `b -= a mod 16` |

初期選択はQFT tabとする。tabは`role="tablist"`、`role="tab"`、`role="tabpanel"`を使い、keyboardで切り替えられるようにする。各tabのsource、execution、入力stateは独立させ、実行中はtab切替を無効化するか、実行中runtimeを安全に停止してから切り替える。

## 2. QFTは固定3-qubitとする

QFT tabの入力は`|000⟩`〜`|111⟩`から選ぶ。UIから任意の重ね合わせは入力させない。

forwardでは3-qubit QFT回路を逐次emitし、8個の複素振幅を更新する。各振幅についてbasis label、絶対値、位相を表示する。QFT後の確率はすべて`1/8`になるため、確率barだけでなく、複素平面上の向きを示すphasorを必須とする。

backwardでは同じJanus procedureを逆方向に実行する。`H`と`SWAP`は同じgateを、controlled phaseは角度の符号を反転したgateをemitする。

```text
H† = H
SWAP† = SWAP
CP(θ)† = CP(-θ)
```

bit orderとQFT回路末尾のswapの有無は実装と数式で統一し、fixture testへ固定する。

## 3. 加算器は固定4-bitとする

MVPは`a`と`b`を4 bitとし、入力範囲を`0`〜`15`に限定する。結果は`a + b mod 16`とする。

bit幅を変えるslider、動的なJanus source生成、任意長registerは最初の実装へ含めない。固定幅にすることで、参照回路の転記、wire配置、全入力検証を小さく保つ。

carry用ancillaは回路が必要とする本数を明示し、すべて`|0⟩`から開始する。forward完了時とbackward完了時にすべて`|0⟩`へ戻ることをUIとtestの両方で検査する。

## 4. 両tabのゲートはhost primitiveとして逐次emitする

静的なforward / backward planを回路図へ一括変換するのではなく、`HostExecutor.call()`と`HostExecutor.uncall()`でprocedureを実行する。

各量子gate primitiveは、forward handlerとbackward handlerからTypeScriptのemitterへ`QuantumGate`を1個ずつ送る。

```ts
type QuantumGate =
  | { kind: "x"; target: QubitId }
  | { kind: "cx"; control: QubitId; target: QubitId }
  | {
      kind: "ccx";
      controls: readonly [QubitId, QubitId];
      target: QubitId;
    }
  | { kind: "h"; target: QubitId }
  | {
      kind: "cp";
      control: QubitId;
      target: QubitId;
      angle: number;
    }
  | { kind: "swap"; targets: readonly [QubitId, QubitId] };
```

primitiveの概念形は次とする。

```ts
const gatePrimitive = (gate: QuantumGate): HostPrimitive<QuantumGate> => ({
  forward: async () => {
    emitter.emit(gate);
    return gate;
  },
  backward: async (receipt) => {
    emitter.emit(inverseGate(receipt));
  },
});
```

`X`、`CNOT`、`Toffoli`、`H`、`SWAP`は自己逆であるため、個々のbackward handlerは同じ論理gateをemitする。`CP(θ)`のbackward handlerだけが`CP(-θ)`をemitする。procedure全体のbackward実行順は`HostExecutor`が反転する。

```text
forward:   G1, G2, G3, ... Gn
backward:  Gn†, ... G3†, G2†, G1†
```

emitterはgateを受け取るたびに次を行う。

1. execution streamへgateを追加する
2. 回路図へgate columnを追加または現在columnをhighlightする
3. 現在のtabに対応するsimulatorへgateを適用する
4. state、register、ancilla、現在stepを描画する
5. 設定された短いdelay後に次のprimitiveへ進む

forwardとbackwardのemit列は各tab内で別のexecutionとして表示し、同じprocedureから逆順かつ各gateのadjointを取った回路が生じたことを確認できるようにする。

## 5. qubit operandと位相角はprimitive名へencodeする

現在のJanus `call`は引数を持たない。本デモのためにparser、AST、resolver、Janus86 compatibility範囲を拡張しない。

qubit IDは合法なidentifierへencodeする。

```janus
procedure add4()
    call cx_a0_b0()
    call ccx_a0_b0_c1()
    call carry_1()

procedure qft3()
    call h_q2()
    call cp_pi_2_q1_q2()
    call cp_pi_4_q0_q2()
```

上記はoperandのencode方法を示す例であり、参照回路の実際のgate sequenceではない。

TypeScript側には、primitive名とgate descriptorを対応させる`QuantumGateCatalog`を置く。

```text
x_b0             → X(target=b0)
cx_a0_b0         → CNOT(control=a0, target=b0)
ccx_a0_b0_c1     → Toffoli(controls=[a0,b0], target=c1)
h_q2              → H(target=q2)
cp_pi_2_q1_q2     → CP(angle=π/2, control=q1, target=q2)
```

複数gateからなる意味のある部分回路は、可能な限りJanusのnested procedureとして表現する。これにより、nested procedure内部の逆順化もJanus runtimeへ委ねられる。

`QuantumGateCatalog`は、未知のwire、同じqubitをcontrolとtargetへ重複指定したgate、未登録gateをcompileまたはregistry構築時に拒否する。

## 6. 入力は計算基底に限定し、tabごとに必要なstateだけをsimulateする

両tabともUIから受け取る入力は計算基底に限定する。

Adder tabはbasis bit列へ次の更新を適用する。

```text
X:        target ^= 1
CNOT:     target ^= control
Toffoli:  target ^= control1 & control2
```

これは論理gate sequenceが計算基底をどの計算基底へ移すかを確認する古典simulationである。

入力register `a`は不変、出力register `b`は`a + b mod 16`、ancillaはzeroというpostconditionをforward完了時に検査する。backward完了時は、`a`、`b`、ancillaを含む全basis stateがforward前と一致することを検査する。

QFT tabはforward後に重ね合わせが生じるため、長さ8の小さなstate vectorを使う。実部と虚部は`Float64Array`または同等の数値配列で保持し、`H`、`CP`、`SWAP`を適用する。各stepでnormが許容誤差内で1であることを検査する。

ブラウザ上で量子hardwareを実行しているとは表示しない。QFT tabは3-qubit state-vector simulation、Adder tabは計算基底に特化した論理gate simulationであることを明記する。

## 7. UIはtabごとの一つの問いへ集中する

ページ共通の主messageは次とする。

> Quantum circuits have a direction. Janus lets you run both.

QFT tabでは次を示す。

* `|x⟩`の10進入力および3-bit表現
* 3本のqubit wireと逐次emitされた`H`、`CP`、`SWAP`
* 8振幅のphasor、絶対値、位相
* `CALL QFT`、`UNCALL QFT`、`RESET`ボタン
* `QFT† QFT |x⟩ = |x⟩`のround-trip結果

Adder tabでは次を示す。

* `a`と`b`の10進入力および4-bit表現
* `a`、`b`、carry ancillaのwire
* 逐次emitされた`X`、`CNOT`、`Toffoli`の回路図
* 現在実行中のgateとstep番号
* `CALL ADD`、`UNCALL ADD`、`RESET`ボタン
* forward / backwardのexecution stream
* `b = a + b mod 16`とround-tripの検証結果
* 参照論文へのlink
* `Logical Toffoli primitives; no Clifford+T decomposition`というscope注記

量子計算、surface code、magic-state distillation、T gate一般を説明する長いtutorialは置かない。論文のtemporary logical-ANDはscope注記または短い補足に留め、加算器の操作を主役にする。

両tabの回路図はbitmap画像ではなく、gate descriptorから生成するinline SVGとする。狭い画面ではwire名を固定し、gate columnsを横scrollできるようにする。

## 実装構成

実装時は次の責務分割を基準とする。正確なfilenameは実装中に調整してよい。

```text
src/quantum/
├── qft-source.ts        固定3-qubit QFTのJanus source
├── adder-source.ts      固定4-bit加算器のJanus source
├── gates.ts             QuantumGate型、catalog、primitive registry
├── basis-simulator.ts   加算器の計算基底simulator
├── state-vector.ts      QFT用の8振幅simulator
├── runtime.ts           HostExecutor、emitter、snapshot、逐次実行
├── html.ts              /quantum HTMLとCSS
└── browser.ts           tab、DOM binding、SVGとphasor描画
```

量子デモruntimeは、既存host runtimeをforkしない。量子固有のgate receipt、emitter、basis stateを薄いadapterとして追加する。

`HostExecutionEvent`はprimitive名、direction、statusを持つためexecution logに利用できる。ただし回路図のsource of truthは文字列化したevent messageではなく、primitiveがemitする型付き`QuantumGate`とする。

## 検証方針

QFTは全8入力、加算器は全256組の`a`と`b`をunit testする。

1. すべての`x ∈ [0,7]`について、QFT後の振幅が定義式と許容誤差内で一致する
2. QFTの各stepでstate-vectorのnormが1である
3. QFT後に逆QFTを適用すると元の`|x⟩`へ戻る
4. QFT backwardのgate順がforwardの逆順で、`CP`の角度だけ符号反転する
5. すべての`a,b ∈ [0,15]`について加算後に`a`が不変である
6. 加算後の`b`が`(a + b) mod 16`である
7. 加算後にすべてのancillaがzeroである
8. 加算結果へbackwardを適用すると全qubitが初期basis stateへ戻る
9. 加算器のbackwardでemitされたprimitive順がforwardの逆順である
10. 各`X`、`CNOT`、`Toffoli`のbackward emitがforwardと同じgate descriptorである
11. nested procedureを含めても両tabで同じround-tripが成立する
12. 不明なgate名、不正なqubit operand、不正なphase角を実行前に拒否する
13. `/quantum`が二つのtabを持つHTMLを返し、量子browser bundleを別pathで返す
14. メインページの既存3デモとbundleが退行しない

数値結果だけでなく、参照回路を転記した期待gate sequenceとの一致をfixture testで固定する。論文の図から回路を転記する際はbit order、最下位bit、carry wire、modulo overflowの扱いをコードとtestに明記する。

## 却下する案

### メインページへ4つ目のtabとして追加する

量子回路固有の前提説明が既存Pure Janusデモの焦点を弱める。別routeにする。

### QFTと加算器を一つのDraper加算器にまとめる

QFT、位相加算、逆QFTを一度に理解する必要があり、Janusのdirection反転よりアルゴリズム説明が大きくなる。QFTとToffoli加算器を独立したtabにし、各tabの見せ場を一つに絞る。

### forward / backward planだけを一括表示する

回路の構造確認には使えるが、Janusからgateが順番に外へ出て状態が変わる体験が弱い。静的planはtestや事前検査に利用してよいが、UIの主経路はhost primitiveの逐次emitとする。

### Pure Janus evaluatorへemit callbackを追加する

Pure Janusの同期的な整数状態変換とhost effectの境界を崩す。量子gate emitは既存host extensionへ置く。

### 本デモのために`call gate(a, b)`を追加する

procedure引数はADR002のJanus86 clean core対象外であり、parserからruntimeまで変更範囲が広い。固定回路ではoperandをprimitive名へencodeすれば足りる。

### 最初からtemporary logical-ANDを忠実に再現する

測定、古典feed-forward、reset、Clifford+T gate、確率的な測定結果の表示が必要になり、加算器と方向反転よりfault-tolerant synthesisの説明が主役になる。将来拡張へ送る。

## トレードオフ

QFT tabを追加すると、複素振幅、phasor、controlled phaseのinverseが必要になり、加算器単独より実装量が増える。一方、すべてのprimitiveが自己逆な加算器だけでなく、`CP(θ)`がbackwardで`CP(-θ)`になる例を見せられる。Janusが単に同じ操作を逆順へ並べるだけでなく、各primitiveの逆方向も合成することが明確になる。

Toffoliをprimitiveにすると、加算回路と逆回路は簡潔になり、計算基底simulationとJanusのdirection反転へ集中できる。一方、参照論文の中心的成果であるtemporary logical-ANDとT-count削減はデモから見えなくなる。この制約は出典の隣に明記し、論文のresource削減をデモの成果として借用しない。

operandをprimitive名へencodeする方法は、一般的な量子programming APIとしては冗長である。ただし固定4-bitデモではsourceとwireの対応が明示的になり、Janus86 coreを拡張せずに済む。可変幅や他回路を増やす段階で、量子回路用IRまたは引数付きhost callを別途検討する。

Adder tabを計算基底限定にすることで、表示結果を古典bit更新として完全に追跡でき、全入力の正しさを小さなtestで検査できる。重ね合わせと位相はQFT tabの固定3-qubit state vectorだけで扱い、一般purpose simulatorへ広げない。

host runtimeのreceipt journalは、単なる回路生成だけなら静的planより重い。しかし逐次animation、gate単位のevent、`call`後の`uncall`、途中状態の検査を既存runtime上で一貫して扱える利点がある。

## Accept条件

1. 量子デモを`/quantum`の別ページとして提供する
2. 量子ページを`QFT`と`Adder`の二つのaccessibleなtabに分ける
3. QFT tabで固定3-qubitの計算基底入力をQFTできる
4. QFT後に8振幅の大きさと位相をphasorで確認できる
5. `uncall qft`で元の計算基底へ戻る
6. QFTのbackwardでgate順が反転し、controlled phaseの角度が反転する
7. Gidneyの論文を加算器の出典としてAdder tabに表示する
8. Toffoliを論理primitiveとして扱い、Clifford+T分解を行わない
9. 論文のT-count / T-depth削減をデモ自身の検証結果として主張しない
