# ADR003: 可逆性をby constructionにするAIコーディング向けデモを定める

## Status

Accepted — Implemented; validation and demo scope under review

## Current state

PR preview environment、AIによる`create_cache`の1行追加、同一sourceからのderived inverse plan、別runtimeでのexecution record再開、receipt、drift検出、途中失敗時のcleanupは実装済みである。

一方、Accept条件8〜10のユーザー検証は未完了である。また、calls-onlyのheadline demoでは「単に後ろからuncallしている」ように見え、実装済みのPure Janusの可逆update、swap、conditional、loopが伝わらないという課題を再レビュ中である。可逆ソート等のPure Janusデモを採用する場合は、本ADRを直接書き換えず、後続ADRでheadline demoの構成を更新する。

## Date

2026-07-18

## 概要

現在のデプロイデモは、forwardに成功した操作のreceiptをstackへ積み、途中失敗時または`Uncall`時に逆順cleanupする。

これはruntimeとUIのsmoke testとしては有効だが、価値の見せ方としては弱い。Pythonの`with` / `ExitStack`、TypeScriptの`using` / `await using` / `AsyncDisposableStack`でも、同一scope内のリソースをLIFO順にcleanupできるためである。

UNCALLの次のheadline demoでは、単なるscope cleanupや永続jobではなく、次の価値を見せる。

> アプリケーション作者は意図する操作列を一度だけ書く。戻す順序とrollback経路は、信頼済みprimitiveとprocedureの構造から機械的に導出する。

特にAIコーディングでは、AIに正しいrollbackコードを毎回生成させるのではなく、rollbackコードを書く自由度そのものを減らす。

人間も、forward、正常終了時cleanup、途中失敗時rollback、partial success、receiptの受け渡しを変更のたびに照合しない。forwardの意図と、新規primitiveがある場合のforward / backward contractへreview対象を絞る。

これはプログラム全体を形式検証する試みではない。**「逆順で元に戻せる」という局所的で価値の高い性質だけを、言語構造とruntimeへ移す部分的な形式手法**として位置付ける。

第一候補を、Pull Requestごとの一時的なpreview environmentとする。

デモでは、AIへ既存のpreview environmentにcacheを追加するよう依頼する。信頼済みの`create_cache` primitiveがすでにある場合、AIがprocedureへ追加するのは`call create_cache()`の1行だけとする。その1行から、正常な`uncall`とすべての途中失敗位置に対するcleanup順序が自動的に更新される。

そのprocedureをPR open時に実行し、構築処理のscopeやprocessが終了しても環境を稼働させる。数時間後または数日後、PR close / mergeを受けた別processがexecution recordを読み込み、同じprocedureを`uncall`して環境を逆順に破棄する。

さらに、作成後に外部変更が入った場合は無条件に削除せず、receiptの世代やpostconditionを検査してcleanupを停止する。これにより、単なるdisposeではなく、時間をまたぐchecked reversalとして見せる。

## ADR000〜ADR002との関係

ADR000の価値仮説である「同じプログラム定義から実行とcleanupを生成する」は維持する。

ADR001 Phase 0のデモも削除しない。Phase 0は次を確認する最小の技術デモとして残す。

* receiptを使って実リソースstateを変更する
* forward成功済みの操作だけを逆順cleanupする
* 途中失敗時に自動rollbackする
* Cloudflare Workersへデプロイできる

ただし、これをUNCALLの最終的な価値説明にはしない。

ADR002のJanus86 clean coreにも変更を加えない。sequenceの反転、`call` / `uncall`の方向反転、静的制約はJanus coreが担う。本ADRが扱うexecution record、host primitive、永続化、drift検査はJanus coreの外側にあるUNCALL effect runtimeの責務とする。

## 背景

### `try/finally`、`with`、`using`ですでにできること

一般的なresource management機構は次を十分に扱える。

* scope exit時のcleanup
* 例外やearly return時のcleanup
* 複数resourceのLIFO cleanup
* 同期または非同期のdispose
* disposerをstackへ動的に追加すること

現在のPhase 0は、これらとほぼ同じ問題を独自runtimeで解いている。

UNCALLが「cleanupを自動化するだけ」のプロダクトなら、言語標準の`using`を選ぶほうが小さく、理解しやすく、信頼しやすい。

### `using`との差を出せる場所

UNCALLが狙うべきなのはresourceの字句的な寿命管理ではなく、**実行された意図を後から反転できる操作単位として残すこと**である。

必要になる性質は次のとおり。

1. forwardを実行したscopeやprocessが終了してもresourceを残せる
2. procedure、receipt、実行順序をexecution recordとして永続化できる
3. 別のeventやprocessがexecution recordを引き継げる
4. 明示的な`uncall`によって、同じprocedureから逆操作列を導出できる
5. 時間経過中の外部変更を検査し、安全でない逆操作を拒否できる
6. 何を、なぜ、どのreceiptで戻したか監査できる

これらは`using`で原理的に実装不可能という意味ではない。disposable objectをscope外へ移譲し、独自に永続化・復元・検査すれば同様の仕組みを作れる。

UNCALLの価値は、その仕組みを個別アプリケーションの慣習ではなく、procedure、direction、receipt、reversibility policyを持つ共通runtimeとして扱うことにある。

### AIコーディングが増やすrollbackの認知負荷

AIへ既存workflowの変更を依頼した場合、通常はforward処理だけでなく、少なくとも次を同時に正しく変更する必要がある。

* 正常終了時のcleanup
* 各途中失敗位置からのrollback
* partial success時に戻す対象
* resource依存関係に対応したcleanup順序
* 作成時に得たIDやversionの受け渡し
* retry時の二重作成・二重削除の回避

AIはこれらを複数箇所へ生成し、人間は生成された経路が互いに整合するかreviewする。forwardへ1操作を追加するだけでも、変更面積とproof obligationが増える。

UNCALLでは、登録済みprimitiveだけを使うアプリケーション変更について、この自由度を除去する。

```diff
 procedure preview_environment()
     call create_namespace()
     call create_database()
+    call create_cache()
     call deploy_application()
```

この変更からruntimeが`delete_cache`を含むinverse planと、各失敗位置に応じたcleanup対象を導出する。AIは別のcleanup workflowを書かず、人間もそれをforward diffと照合しない。

### 部分的な形式手法としての保証境界

「書きさえすれば戻せる」は、任意の外部effectが自動的に可逆になるという意味ではない。

保証を二層に分ける。

#### 構造から保証する層

* statement sequenceを逆順に実行する
* backwardでは`call`と`uncall`を交換する
* forward成功済みのprimitiveだけをcleanup対象にする
* forwardが返したreceiptを対応するbackward handlerへ渡す
* procedure変更に応じてinverse planを自動更新する
* backward handlerを持たないprimitiveをlink時に拒否する

これらはアプリケーションごとにAIやreviewerが再確認しない。

#### 信頼境界としてreviewする層

* primitiveのforward / backwardが正しい対応であること
* receiptが逆操作に十分な情報を持つこと
* exact、checked、compensating、irreversibleの分類が妥当であること
* 外部stateのdriftをどの条件で拒否するか

新規primitiveを追加する場合、この層のreviewとround-trip testは必要になる。一度信頼済みcatalogへ登録したprimitiveをprocedureから使う限り、アプリケーション作者は逆操作を毎回書かない。

したがってUNCALLが減らすのは、すべてのcorrectness reviewではなく、**合成のたびに繰り返されるrollback順序と経路のreview**である。

## 価値仮説

次のデモを見た人は、UNCALLのAIコーディング上の価値を一文で説明できる。

> AIがforward procedureへ1行追加すれば、信頼済みprimitiveからrollback経路も更新される。人間は別のcleanup diffをreviewしなくてよい。

`using`との差は次の一文で説明できる。

> `using`はscopeが所有する値の寿命を管理する。UNCALLは、書かれた操作列からinverse planを作り、後日の別eventでも実行できる。

価値仮説をさらに分解する。

* AIがforwardとcleanupを別々のworkflowとして生成・保守しなくてよいことに価値がある
* 人間が複数のrollback経路をforward diffと照合しなくてよいことに価値がある
* procedureへ操作を追加するとinverse planもby constructionに更新されることに価値がある
* 信頼すべき範囲を小さなprimitive catalogへ限定できることに価値がある
* execution recordがprocessをまたいで引き継げることに価値がある
* 逆操作列を実行前に表示・検査できることに価値がある
* 外部stateが変化していれば停止するchecked reversalに価値がある
* rollbackが特別な失敗経路ではなく、明示的に何度も試せる通常機能になることに価値がある

## 第一候補: Pull Request Preview Environment

### シナリオ

Act 1でAIがcacheを追加した後、PRがopenされたとき、UNCALLが次のprocedureから専用のpreview environmentを構築する。

```janus
procedure preview_environment()
    call create_namespace()
    call create_database()
    call create_cache()
    call seed_preview_data()
    call deploy_application()
    call attach_preview_url()
```

forward handlerは、各resourceのIDだけでなく、逆操作時の検査に必要な情報をreceiptへ入れる。

```ts
type Receipt = {
  resourceId: string;
  generation: number;
  createdFor: string;
  postconditionHash: string;
};
```

PR openを処理したprocessは、procedure名、sourceまたはplan hash、receipt stack、PR番号をexecution recordとして保存して終了する。この時点ではcleanupしない。

```text
Session A / PR opened
  call preview_environment
  → preview URL is live
  → execution exec_7F3 saved
  → Session A ends

12 hours later

Session B / PR merged
  load exec_7F3
  uncall preview_environment
  → detach_preview_url
  → undeploy_application
  → unseed_preview_data
  → delete_cache
  → delete_database
  → delete_namespace
```

### このシナリオが適している理由

* resourceをscope exit後も残すことが要件なので、`with` / `using`との差が自然に出る
* PR openとPR mergeが別eventであり、process境界を説明しやすい
* preview environmentは一時的で、最終的に片付ける意図が明確である
* 作成順と依存関係を逆にした破棄順が直感的に分かる
* 実クラウドAPIを使わなくても、時間軸と永続execution recordをモックで表現できる
* 将来GitHub webhookや実クラウドへ接続する拡張経路が分かりやすい

### 見せ場

デモは次の4幕にする。

#### Act 1: AIがforwardへ1行だけ追加する

最初のprocedureにはcacheがない。ユーザーがAIへ「Applicationの前にpreview用cacheを追加して」と依頼する。

`create_cache`が信頼済みprimitive catalogにある場合、AIはprocedureへ`call create_cache()`を1行追加する。画面はsource diffと同時に、生成されたinverse planの差分を表示する。

```diff
 forward                         backward derived by runtime
+ call create_cache()           + delete_cache(receipt)
```

AIは正常cleanup、途中失敗rollback、receipt stackのコードを編集しない。reviewerが確認するapplication diffも1行とする。

`create_cache`がcatalogにない場合は、backward handler未登録のlink errorにする。「書けば必ず戻せる」という保証を、未検証primitiveを暗黙に生成することで満たさない。

#### Act 2: 別の日まで残す

`PR opened`を押すとpreview environmentが構築され、execution IDが発行される。

画面上で「forward process ended」を明示する。scopeが終了してもresourceが残り、`using`のように自動disposeされないことを見せる。

#### Act 3: 別processが引き継ぐ

`PR merged — new process`を押すと、新しいruntime instanceが永続化済みexecution recordを読み込む。

実行前に、procedureから導出した逆操作列と、各操作が使うreceiptを表示する。ユーザーが確認して`Uncall`すると逆順cleanupする。

#### Act 4: 時間経過によるdriftを検出する

`Simulate external database change`を有効にすると、databaseのgenerationまたはpostcondition hashがreceiptと一致しなくなる。

`Uncall`はApplicationとCacheまでcleanupした後、Databaseの削除を拒否し、Namespaceも依存関係上残す。

画面には次を表示する。

```text
delete_database blocked
expected generation: 4
current generation: 5
manual decision required
```

「何でも元に戻せる」と主張せず、checked reversalの境界を見せること自体を価値にする。

## `using`との比較をデモ内でどう伝えるか

説明文だけで比較せず、時間軸をUIに置く。

| | `with` / `using` | UNCALL preview demo |
| --- | --- | --- |
| アプリケーション作者が書くもの | resource取得とdisposable valueの保持 | forwardのprocedure |
| 操作追加時の変更 | disposerの所有・scopeも確認 | 信頼済みprimitiveなら`call`を1行追加 |
| reviewerが照合するもの | 取得、所有権、dispose経路 | forwardの意図。primitive追加時だけpairもreview |
| cleanupの契機 | scope exit | 後日の明示的event |
| cleanupする主体 | 同じ実行context | execution recordを受け取った別process |
| 保持するもの | live disposable value | procedure identityとserializable receipt |
| 逆操作の組み立て | disposerをresourceごとに登録 | procedureの構造から逆順を導出 |
| 時間経過中の変更 | disposer固有の実装 | reversibility policyとして検査・表示 |
| 主目的 | resource lifetime management | durable, inspectable reversal of intent |

デモ中の短い説明は次とする。

> `using` closes what this scope owns. UNCALL reverses what a previous event did.

この表現が技術的に強すぎる場合は、次を使う。

> UNCALL turns a completed operation into a durable, inspectable undo plan.

AIコーディングの文脈では、次を主messageにする。

> Do not ask AI to write every rollback path. Make rollback a property of the program structure.

### 第二候補: Incident Mitigation

障害時にtrafficをmaintenance backendへ切り替え、feature flagを無効化し、暫定capacityを追加する。障害解消後、同じmitigation procedureを`uncall`する。

これは「適用workflow」と「解除workflow」のdriftを防ぐ価値を強く示せる。一方で、安全性への期待が高く、モックでは誇張に見えやすい。preview environmentでexecution recordとchecked reversalを確立した後の第二デモとする。

### 補助候補: Feature Flag Rollout

複数tenantへ順番にflagを適用し、一部失敗時に成功済みtenantだけを戻す。実装は小さいが、逆順である必然性とreceiptの価値が弱い。

headline demoではなく、UNCALL effect runtimeをアプリケーション操作へ適用できることを示す短い例に向く。

## デモの成功条件

技術的な完了だけでなく、価値が伝わったかを条件にする。

1. 初見のユーザーが90秒以内に「AIがrollback経路を書かなくてよい」と説明できる
2. 信頼済みprimitiveをprocedureへ1行追加すると、inverse planが自動更新される
3. 操作追加時に、正常cleanupと各途中失敗位置のためのapplication codeを追加しない
4. backward handlerが未登録のprimitiveは実行前にlink errorになる
5. reviewerへforward diffとderived inverse planを並べて表示できる
6. 初見のユーザーが`using`との違いも説明できる
7. forwardを実行したruntime instanceを破棄しても、別instanceから`uncall`できる
8. execution recordにprocedureまたはplan hash、receipt、実行順序が含まれる
9. 逆操作列が別の手書きcleanup planではなく、同じprocedureから導出される
10. 正常時は全resourceが逆順に消える
11. drift発生時は危険なcleanupを拒否し、残ったresourceと理由を表示する
12. どのreceiptを使って各backward handlerを実行したか確認できる
13. 実クラウドAPIを使わない場合、mockであることをUIに明記する
14. 「厳密な逆操作」と「条件付きcleanup」を混同しない
15. `using`でも同等機構を組めることを否定せず、UNCALLが共通化する部分を説明する

ユーザーテストでは、デモ後に次の質問をする。

* AIが通常書かなくてよくなったコードは何だと思うか
* 人間がreviewしなくてよくなった組み合わせは何だと思うか
* それでも人間がreviewすべき境界はどこだと思うか
* receiptはなぜ必要だと思うか
* 自動で戻してほしくないのはどのような場合か
* 自分の仕事で「後日の別eventから戻したい操作」があるか

最後の質問に具体例が出なければ、技術的に理解されても価値仮説は弱いと判断する。

## 実装方針

本ADRのaccept前に、実クラウド統合は行わない。まずbrowserまたはWorker内のmockでstoryboardを検証する。

必要な最小追加は次のとおり。

* forward source diffとderived inverse plan diffの並列表示
* forward / backwardの両方を要求するprimitive manifestとlink error
* primitive pairのround-trip test
* procedure内の各失敗位置に対する自動cleanup test
* serializableなexecution record
* execution IDによる保存と読込
* runtime instanceを作り直してからのresume
* plan hashまたはprocedure version
* receiptごとのprecondition / postcondition検査
* cleanup blocked状態
* 実行前のinverse plan表示

永続化先をCloudflare KV、D1、Durable Objects、browser storageのどれにするかは本ADRでは決めない。デモのstoryboardと必要な整合性を確認した後、別ADRで選定する。

Phase 2 evaluator統合との順序は次とする。

1. ADR001 Phase 2を完了し、表示sourceと実行procedureを一致させる
2. primitive manifestでforward / backward pairをlink時に検査する
3. `call`を1行追加したときのderived inverse plan差分を表示する
4. 各失敗位置のcleanupを自動生成し、property testする
5. 同一browser session内でexecution recordをserialize / restoreする
6. 新しいruntime instanceから`uncall`する
7. process境界を表現するUIへ変更する
8. 必要ならWorker側の永続storageへ移す

## 却下する見せ方

### 「例外が起きても自動cleanupできる」を主役にする

`try/finally`、`with`、`using`ですでに解ける。UNCALL runtimeの正しさを示すテストとしては残すが、headlineにはしない。

### `using`では不可能だと主張する

不正確である。disposable stackを独自に永続化・移譲すれば同様の結果は作れる。

### 何でも完全に元へ戻せると主張する

外部effectは時間経過、第三者による変更、不可逆な通知などを含む。drift検査で停止する挙動を隠さない。

### 実クラウドを先に接続する

価値仮説よりAPI統合、認証、cleanup事故対策の実装が大きくなる。mockでstoryを検証してから判断する。

### 永続化だけを差別化にする

永続jobやsaga engineにも同じ機能がある。UNCALLの特徴は、同じprocedureのdirectionを反転し、逆操作列を構造から導出する点と組み合わせて初めて出る。

### 「AIが書けばどんなeffectでも自動的に戻せる」と主張する

新しいhost primitiveのforward / backward pairは信頼境界であり、reviewとtestが必要である。減らすのはapplicationごとに繰り返す合成とrollback経路のproof obligationであって、外部APIの意味を自動証明するものではない。

### AIにcleanup codeも生成させ、その正しさを別のAIにreviewさせる

生成とreviewを増やすだけでは認知負荷と不整合の自由度が残る。UNCALLでは、導出可能なcleanup codeをそもそもapplication sourceへ書かせない。

## トレードオフ

このシナリオはPhase 0より実装量が増える。primitive catalog、link contract、inverse plan表示に加えて、execution recordのschema、procedure version、partial cleanup、drift、resumeを扱う必要がある。

可逆性の保証も、信頼済みprimitiveの正しさより外側へは広がらない。primitive catalogが大きくなれば、そのreview、versioning、contract testが新しい負担になる。ただし、この負担はapplicationの各利用箇所ではなくprimitive定義へ集約できる。

また、実用上はTemporalなどのdurable execution engine、Infrastructure as Codeのdestroy plan、Sagaのcompensation、workflow engineとの比較が必要になる。`using`との差だけを説明しても、プロダクトとしての独自性は証明できない。

一方で、この複雑さを避けるとUNCALLは言語標準のresource managementに対する薄い構文糖に見える。AIが変更するapplication codeからrollback経路を除去できること、さらにそのinverse planを時間とprocessをまたいで扱えることは、UNCALLを独立した言語・runtimeとして作る意味を検証するために必要である。

## Accept条件

次を満たした場合に本ADRをAcceptedとする。

1. Phase 0デモを技術smoke testとして残し、headline demoから降格することに合意する
2. 中心価値を「AIと人間がapplicationごとのrollback経路を生成・reviewしなくてよいこと」とする
3. 保証対象をprocedureの構造的反転と信頼済みprimitiveの合成に限定し、primitive pair自体は信頼境界と明記する
4. PR preview environmentを次の価値検証シナリオとして採用する
5. AIが信頼済みprimitiveの`call`を1行追加し、inverse planが自動更新されるstoryを含める
6. `using`との差を「不可能／可能」ではなく「scope lifetime／reversible-by-constructionな操作列／durable reversal intent」と説明する
7. drift時に自動cleanupを停止するstoryを含める
8. storyboardを少なくとも3人に見せ、2人以上がAIとreviewerの認知負荷削減を自分の言葉で説明できる
9. 同じ2人以上が`using`との差も自分の言葉で説明できる
10. そのうち1人以上から、自分の業務にある類似操作の具体例が得られる

この条件を満たせない場合、execution persistenceの実装へ進まず、別シナリオまたはプロダクト仮説を再検討する。
